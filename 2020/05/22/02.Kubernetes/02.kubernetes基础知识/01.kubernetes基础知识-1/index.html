01.kubernetes基础知识-1

HeiYueQiMa 2020-05-22 07:38:25
Kubernetes

一. 入门知识简介

文章摘录k8s官方文档

1. k8s版本信息

Kubernetes 版本号格式为 x.y.z，其中 x 为大版本号，y 为小版本号，z 为补丁版本号。

  • kube-apiserver

    1. 多个apiserver版本最多相差1 为了兼容升级

  • kubelet

    1. kubelet 最多只能比apiserver低一个斑斑
    2. kubelet 版本取决于最低的apiserver版本

  • kube-controller-manager, kube-scheduler, and cloud-controller-manager

    1. 最好版本是kube-apiserver版本一致
    2. 为了支持在线升级 所以兼容低一个小版本

  • kubectl

    1. 可以比apiserver低一个版本也可以高一个版本

2. k8s组件升级顺序

升级之前必须确保版本都满足条件

  1. apiserver
  2. kube-controller-manager,kube-scheduler
  3. kubelet

3.容器运行时

就是container runtime interface

  • Cgroup 驱动程序

如果linux的发行版使用systemd作为初始化系统时,会有2个cgroup
1. 尽量使用systemd作为cgroup

2. 使用cgroupfs 系统 没有 systemd稳定

3. native.cgroupdriver=systemd 设置

不要更改已经加入集群节点的cgroup驱动

  • docker 安装

这个我一般是用docker官方的脚本一件安装
设置参数可以保存一下

这里推荐的参数 使用systemd作为cgroud

使用了更稳健的overlay2 

{
  "exec-opts": ["native.cgroupdriver=systemd"],
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "100m"
  },
  "storage-driver": "overlay2",
  "storage-opts": [
    "overlay2.override_kernel_check=true"
  ]
}

4.构建大型集群

在1.18版本中 k8s最大节点数量为5000。 详细配置

  • node 数量不超过5000
  • pod总数不超过150000
  • 容器总数不超过300000
  • 每个节点的pod不超过100
  1. 插件资源
    为了防止内存泄露或者集群插件的其他问题导致节点可用资源被消耗
    k8s限制了插件容器可以消耗的cpu和内存资源

在大型规模集群上运行时 插件会消耗大量资源 如果不调整这些值的情况下部署了大规模集群 插件容器可能会由于达到限制而不断的被杀死

创建大规模集群时候,注意这些程序

  • influxDB 和 Grafana
  • kubedns dnsmasq 和 sidecar
  • Kibana
  • FluentD 和 ElasticSearch 插件
  • FluentD 和 GCP 插件

根据集群规模调整副本数量 有助于增加处理的负载

  • elasticsearch

综合就是大规模集群 考虑插件已经日志监控等服务的资源消耗情况 做适当的调整

5.校验节点设置

  1. 节点的前提条件

    必须有 kubelet 和 cri(docker)

  2. 节点一致性

    kubernetes为硬件提供了节点一致性的测试docker镜像

    arch image
    amd64 node-test-amd64

    这个讲实话 文档写的没明白

6.pki证书和要求

Kubernetes 需要 PKI 证书才能进行基于 TLS 的身份验证。

不将私钥存储在api服务器上 可以让私钥更加安全

  1. 集群是如果使用证书的

    kubernetes需要PKI才能执行的操作

    • kubelet的客户端证书 用于与api服务器的身份验证
    • apiserver的端点证书
    • 集群管理员的客户端证书 用于与api服务器的身份验证
    • apiserver的客户端证书 用于和kubelet的会话
    • apiserver的客户端证书 用于和etcd的会话
    • 控制器管理器的客户证书/kubeconfig 用于和apiseever的会话
    • 调度器的客户端证书/Kubeconfig 用于和apiserver的会话
    • kube-proxy的客户端以及服务端的证书
    • etcd 还实现了双向 TLS 来对客户端和对其他对等节点进行身份验证。

  2. 证书存放的位置

    如果是kubeadm安装的话 证书存放在/etc/kubernetes/pki 里面

  3. 自己生成证书

    这个适用于kubeadm的证书生成 我是用的二进制安装 就不列了

k8s概念部分

7. 概述

要使用k8s的话，你需要用k8s的api来描述集群的预期状态，包括你运行的应用、负载，以及他们使用的镜像、副本数，还有所需的网络和磁盘资源。一旦你设置了目标状态 k8s 控制器就会自动执行各类任务，比如说重启容器，调整副本数之类的操作。

  • k8s master 组件

    1. kube-apiserver
    2. kube-controller-manager
    3. kube-scheduler

  • k8s work 组件

    • kubelet
    • kube-proxy
  1. k8s 对象

k8s包含若干用来表示系统状态的抽象层。这些抽象使用k8s api对象来表示

基本的k8s对象包括

  • pod
  • service
  • volume
  • namespace

k8s还包含大量被称作controller的高级抽象。控制器基于基本的对象构建并提供额外的功能和方便使用的特性

  • deployment 无状态服务
  • daemonset 守护服务
  • statefulset 有状态服务
  • replicaset 副本集
  • job
  1. kubectl 默认命名空间
# 指定默认命名空间 
kubectl config set-context --current --namespace=xxx
# 查看哪些资源属于命名空间
kubectl api-resources --namespaced=true

  1. k8s常用标签

    • “release” : “stable”, “release” : “canary”
    • “environment” : “dev”, “environment” : “qa”, “environment” : “production”
    • “tier” : “frontend”, “tier” : “backend”, “tier” : “cache”
    • “partition” : “customerA”, “partition” : “customerB”
    • “track” : “daily”, “track” : “weekly”

这些只是常用标签的例子; 您可以任意制定自己的约定。请记住，对于给定对象标签的键必须是唯一的。
